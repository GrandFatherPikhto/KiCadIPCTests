# KiCad IPC Test Framework

**Общее назначение проекта** – создание унифицированной, безопасной и расширяемой системы для тестирования и автоматизации взаимодействия с PCB-редактором **KiCad** через IPC (библиотека `kipy`). Инструментарий позволяет:

- выполнять **дымовые, безопасные и мутирующие тесты** на реальной плате;
- проводить **статический контрактный анализ** установленной версии `kipy`;
- автоматизировать типовые операции (перемещение, переворот компонентов, создание переходных отверстий);
- диагностировать состояние соединения и воспроизводить известные баги KiCad.

Проект построен вокруг **единого раннера** (`runner`), который управляет конфигурацией, логированием, реестром тестов и соединением с KiCad, обеспечивая целостность и повторяемость прогонов.

---

## Установка и зависимости

- **Python 3.8+**
- **KiCad 9 или 10** с запущенным GUI (IPC работает только с открытым PCB-редактором)
- **Библиотека `kipy`** – официальная Python-обёртка для IPC KiCad (устанавливается отдельно, см. [репозиторий](https://gitlab.com/kicad/libraries/kipy))
- Дополнительно для экспорта netlist через CLI – **`kicad-cli`** (входит в состав KiCad)

Рекомендуется использовать виртуальное окружение:

```bash
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install kipy
```

## Конфигурирование

Все настройки выполняются через **единый YAML-файл**. Минимальная структура:

```yaml
kicad:
  timeout_ms: 20000               # таймаут IPC-вызовов (мс)

logging:
  console_level: INFO             # уровень для консоли
  file: logs/test.log             # путь к лог-файлу
  file_level: DEBUG               # уровень для файла
  rotate_max_bytes: 5242880       # размер одного файла
  rotate_backups: 3               # количество ротаций

boards:                           # профили параметров для разных плат
  10CL006:
    ref: "C5"                     # refdes по умолчанию
    pad: "1"                      # номер пада
    zone: "RA_DECAP_ZONE"         # имя зоны
    net: "GND"                    # имя цепи
    extra:                        # дополнительные параметры
      offset_mm: 1.2

suites:                           # группы тестов
  smoke:
    enabled: true
    tests:
      smoke_footprints:
        enabled: true
  mutating:
    enabled: false                # опасные тесты по умолчанию выключены
    tests:
      mutating_flip_then_update_items:
        enabled: false
        dangerous: true
        params:
          ref: "Ctest"
```

Параметры из `boards` и `tests` объединяются через `resolve_params` (параметры теста имеют приоритет).

## Запуск

**Основная команда:**

```bash
python -m runner.run --config <config.yaml> [--board <profile>] [--suite <suite>] [--test <test>]
```

- `--config` – обязательный, путь к YAML-конфигу.
- `--board` – имя профиля платы (подставляет параметры `ref/pad/zone/net`).
- `--suite` – запустить только указанную суиту.
- `--test` – запустить один конкретный тест (даже если он не описан в конфиге – выполнится с параметрами по умолчанию, **кроме опасных** – они требуют явного `enabled: true`).

**Примеры:**

```bash
# Все безопасные тесты для платы 10CL006
python -m runner.run --config config/10CL006.yaml --board 10CL006 --suite safe

# Одиночный дымовой тест на компонент C5
python -m runner.run --config config/10CL006.yaml --board 10CL006 --test smoke_footprints

# Опасный тест (только если включён в конфиге)
python -m runner.run --config config/10CL006.yaml --test decap_create_one_via
```

После выполнения раннер выводит сводку `[PASS]/[FAIL]` и код завершения: `0` – все успешно, `1` – были ошибки, `2` – попытка запустить выключенный опасный тест.

## Важные замечания

- **Опасные тесты** (флаг `dangerous: true`) **никогда не запускаются** без явного `enabled: true` в конфиге – это защита от случайного повреждения платы.
- **Единое соединение** – все тесты в одном прогоне используют одно и то же подключение, что предотвращает утечки сокетов.
- **После GUI-флипа** обязательно перечитывайте плату, иначе локальные объекты устареют.
- **Статические тесты** не требуют запущенного KiCad и могут выполняться в CI.

## Расширение

Чтобы добавить новый тест:

1. Создайте файл в одном из подкаталогов `tests/`.
2. Определите функцию с сигнатурой `def run_test(logger, **params) -> bool`.
3. Зарегистрируйте её декоратором `@register("unique_name", suite="my_suite", dangerous=False, needs_kicad=True)`.
4. Опишите тест в YAML-конфиге (с параметрами при необходимости).

Раннер автоматически обнаружит тест при импорте пакета `tests`.

Проект разработан для внутреннего использования и тестирования IPC-возможностей KiCad. Все исходные коды распространяются под лицензией, совместимой с KiCad (GPLv3).

# Модуль `runner` — единая система запуска тестов и скриптов

`runner` — это каркас для организации, конфигурирования и выполнения тестовых сценариев, работающих с KiCad через IPC. Он обеспечивает:

- **Единую точку входа** — скрипт `run.py` с аргументами командной строки.
- **Гибкую конфигурацию** через YAML-файл, где можно включать/отключать тесты, задавать параметры для разных плат и указывать опасные тесты.
- **Реестр тестов** на основе декораторов — тесты регистрируются автоматически при импорте.
- **Единое подключение к KiCad** на весь прогон, чтобы избежать утечек сокетов и ложных ошибок `busy`.
- **Стандартизированное логирование** с выводом в консоль и файл с ротацией.
- **Обработку ошибок и исключений** — тест считается проваленным при любом исключении; раннер перехватывает и логирует.
- **Защиту от случайного запуска опасных тестов** — они выполняются только при явном `enabled: true` в конфиге, даже если указаны через `--test`.

## Структура модуля `runner`

### 1. `config_schema.py` — модели данных и загрузка конфигурации

Определяет dataclass’ы для всех уровней конфига:

- **`KicadConfig`** — настройки IPC: `timeout_ms` (по умолчанию 20000), `socket_path` (опционально).
- **`LoggingConfig`** — уровни логирования, путь к файлу, ротация.
- **`BoardProfile`** — именованный набор параметров для конкретной платы: `ref` (refdes компонента по умолчанию), `pad` (номер пада), `zone` (имя зоны), `net` (имя цепи), а также `extra: Dict` для любых дополнительных параметров.
- **`TestEntry`** — описание одного теста: `name`, `enabled` (bool), `dangerous` (bool), `params` (словарь переопределяющих параметров).
- **`SuiteConfig`** — группа тестов: `name`, `enabled`, `tests: Dict[str, TestEntry]`.
- **`RootConfig`** — корневой узел, содержащий `kicad`, `logging`, `boards` (словарь профилей) и `suites`.

Функции:
- `load_config(path)` → `RootConfig` — загружает и парсит YAML.
- `resolve_params(board_profile, test_entry)` → `Dict[str, Any]` — собирает итоговые параметры для теста: сначала поля `ref/pad/zone/net` из профиля платы, затем `extra` из профиля, затем `params` из записи теста (побеждают при конфликте).

---

### 2. `logging_setup.py` — настройка логирования

- `setup_logging(logger_name, console_level, file_path, file_level, rotate_max_bytes, rotate_backups)` → `Logger`.
- Создаёт два обработчика: консольный (уровень `console_level`, обычно `INFO`) и файловый с ротацией (уровень `file_level`, обычно `DEBUG`).
- **Идемпотентен** — при повторном вызове с тем же именем логгера старые обработчики удаляются, чтобы не плодить дубли.
- Отключает пропагацию в корневой логгер (`logger.propagate = False`).

---

### 3. `registry.py` — реестр тестов

- Глобальный словарь `_REGISTRY: Dict[str, RegisteredTest]`.
- **`RegisteredTest`** — dataclass с полями: `name`, `suite`, `dangerous`, `needs_kicad`, `func`.
- Декоратор **`@register(name, suite, dangerous=False, needs_kicad=False)`** регистрирует функцию. Проверяет уникальность имени.
- `needs_kicad=True` означает, что тесту нужны объекты `kicad` и `board`; раннер передаст их через параметры (из `SharedConnection`).
- Функции доступа:
  - `get(name)` — получить зарегистрированный тест.
  - `all_tests()` — все тесты.
  - `tests_in_suite(suite)` — тесты конкретной суиты.
  - `clear()` — сбросить реестр (для тестирования самого раннера).

---

### 4. `step_helper.py` — пошаговое логирование с таймингом

- **`call_step(logger, label, fn, *args, **kwargs)`** → `(result, success)`.
- Вызывает `fn`, логирует метку, время выполнения в миллисекундах.
- При исключении логирует ошибку и возвращает `(None, False)` — **не бросает исключение**.
- Используется в тестах для детального отслеживания каждого вызова API, чтобы понимать, на каком шаге произошла ошибка.

---

### 5. `run.py` — главная точка входа

**Основной скрипт**, выполняющий следующие шаги:

#### Парсинг аргументов командной строки

- `--config` (обязательный) — путь к YAML-конфигу.
- `--board` (опционально) — имя профиля платы из конфига.
- `--suite` (опционально) — запустить только одну суиту.
- `--test` (опционально) — запустить только один тест по имени.

#### Инициализация

1. Загружает конфиг (`load_config`).
2. Настраивает логирование через `setup_logging` с параметрами из конфига.
3. Выполняет **дискавери тестов**: функция `discover_tests(package_name="tests")` рекурсивно импортирует все модули внутри пакета `tests`. Это гарантирует, что все декораторы `@register` сработают и наполнят реестр.

#### Отбор тестов для запуска

Функция `select_tests(cfg, suite_name, test_name)` возвращает список `(suite_name, TestEntry)` для запуска с учётом `enabled` и `dangerous`:

- Если указан `--test`:
  - Ищет тест во всех суитах конфига.
  - Если найден и `enabled: false` — выбрасывает `PermissionError` (защита).
  - Если теста нет в конфиге — создаёт временный `TestEntry` с `enabled=True`, `dangerous=False`.
- Если `--suite` указан — берёт только эту суиту (если она `enabled`).
- Иначе — все суиты из конфига.
- Внутри суиты проходится по всем тестам, проверяя `enabled` (и для опасных — что `enabled` явно `true`).

#### Управление соединением

Класс **`SharedConnection`**:

- Лениво устанавливает соединение с KiCad через `kicad_client.connect()` при первом обращении.
- Предоставляет метод `get()` → `(kicad, board)`.
- Метод `refresh_board()` перечитывает плату (полезно после GUI-действий).
- Используется для всех тестов с `needs_kicad=True`, чтобы открыть **одно** соединение на весь прогон.

#### Запуск тестов

Для каждого отобранного теста:

1. Получает `RegisteredTest` из реестра по имени.
2. Формирует параметры через `resolve_params(board_profile, entry)`.
3. Если `needs_kicad`, добавляет `kicad` и `board` из `SharedConnection`.
4. Логирует запуск (с пометкой `[ОПАСНЫЙ]`, если нужно).
5. Вызывает функцию теста с параметрами.
6. Перехватывает любые исключения, логирует их и считает тест проваленным.
7. Сохраняет результат `(имя, ok)`.

#### Итоговый отчёт

- Логирует общее количество пройденных тестов.
- Выводит каждый тест с `[PASS]`/`[FAIL]`.
- Завершается с кодом `0` (все успешно) или `1` (хотя бы один упал).
- Специальный код `2` — если `--test` указан на выключенный тест (защита).

## Важные архитектурные решения

- **Безопасность по умолчанию** — опасные тесты (`dangerous: true`) не запускаются, даже если указан `--test`, если в конфиге не стоит `enabled: true`. Это защита от случайного повреждения платы.
- **Единое соединение** — все тесты в одном прогоне используют одно и то же подключение, что предотвращает ошибки `busy` и утечки ресурсов.
- **Идемпотентность логирования** — повторные вызовы `setup_logging` не плодят обработчики.
- **Прозрачная обработка ошибок** — исключения в тестах не прерывают весь прогон, а логируются; раннер продолжает выполнение остальных тестов.

---

## Расширение и добавление новых тестов

1. Создайте Python-файл в любом подкаталоге `tests/` (например, `tests/my_suite/test_my_feature.py`).
2. Определите функцию с сигнатурой `def run_test(logger, **params) -> bool`.
3. Зарегистрируйте её декоратором:
   ```python
   from runner.registry import register

   @register("my_test", suite="my_suite", dangerous=False, needs_kicad=True)
   def run_test(logger, kicad, board, ref=None, **params):
       # тело теста
       return True
   ```
4. Опишите тест в YAML-конфиге (указав `enabled`, `params` при необходимости).
5. Запустите с `--suite my_suite` или `--test my_test`.


# core_api — базовый слой взаимодействия с KiCad

`core_api` — это универсальная библиотека-обёртка над официальной Python-библиотекой `kipy` (KiCad IPC). Она предоставляет единый, типобезопасный и документированный интерфейс для работы с открытой платой в PCB-редакторе KiCad. 

### Основные принципы

- **Отсутствие глобального состояния** — все функции принимают готовые объекты `kicad` или `board` как явные аргументы. Это делает код легко тестируемым, переиспользуемым и безопасным в многопоточных сценариях.
- **Транзакционный подход** — все изменения платы выполняются внутри коммитов (`begin_commit` / `push_commit` / `drop_commit`). Это гарантирует атомарность и позволяет откатить изменения в случае ошибки.
- **Работа в миллиметрах** — все координаты и размеры принимаются и возвращаются в миллиметрах (тип `float`). Внутреннее представление `kipy` использует нанометры (целые числа), конвертация выполняется автоматически с помощью константы `MM = 1_000_000`.
- **Явное обновление кеша** — после операций, изменяющих плату вне стандартного механизма `update_items` (например, после GUI-флипа через `run_action`), локальные объекты становятся неактуальными. Рекомендуется перечитывать данные через `board.get_footprints()` или `board.refresh()`.

---

## Структура модулей

### 1. `geometry.py` — геометрические константы и утилиты

Самый базовый модуль, определяющий единицу перевода координат:

```python
MM = 1_000_000  # количество нанометров в одном миллиметре
```

Вспомогательные функции:

- `vec_mm(x_mm, y_mm)` → `Vector2` — создаёт вектор из миллиметров.
- `to_mm(vector)` → `(x_mm, y_mm)` — преобразует вектор в миллиметры.
- `bbox_size_mm(bbox)` → `(width_mm, height_mm)` — извлекает размеры из объекта `Box2`.

Все остальные модули опираются на этот модуль для согласованного перевода единиц.

---

### 2. `kicad_client.py` — подключение, управление сессией

Отвечает за установку соединения с запущенным экземпляром KiCad и выполнение низкоуровневых действий:

- `connect(timeout_ms=20000)` → `KiCad` — подключается к текущему открытому проекту. Таймаут по умолчанию увеличен до 20 секунд (в `kipy` по умолчанию 2 секунды, что недостаточно для тяжёлых транзакций).
- `get_board(kicad)` → `Board` — возвращает объект платы.
- `run_action(kicad, action)` — запускает GUI-действие (например, `"pcbnew.InteractiveEdit.flip"`). Это официально нестабильный API (см. документацию `kipy`), но он необходим для операций, недоступных через `update_items` (например, настоящий переворот компонента с зеркалированием).
- `get_version(kicad)` → `str` — возвращает версию KiCad (полезно для диагностики).

---

### 3. `board.py` — транзакции, коммиты и метаданные платы

Самый важный модуль для изменения платы. Он предоставляет:

- **Управление транзакциями**:
  - `begin_commit(board)` — начинает транзакцию.
  - `push_commit(board, commit, description)` — фиксирует изменения.
  - `drop_commit(board, commit)` — откатывает транзакцию.
  - `commit_with_retry(board, description, work_fn, retries=1)` — выполняет переданную функцию внутри транзакции с автоматическим откатом при исключении и одним повторным вызовом. Возвращает `bool` (успех/неуспех) или бросает исключение, если все попытки провалились.

- **Обновление кеша**:
  - `refresh(kicad)` → `Board` — перечитывает плату заново. **Обязательно** вызывать после GUI-действий (например, `flip_selected`), чтобы получить актуальные локальные объекты.

- **Метаданные платы**:
  - `get_edge_bounding_box_mm(board)` → `(width_mm, height_mm)` или `None` — размеры по контуру Edge.Cuts.
  - `get_copper_layer_count(board)` → `int` или `None` — количество медных слоёв.
  - `get_info(board)` → `dict` — объединяет все метаданные в один словарь.

**Важно:** при написании `commit_with_retry` переменная `commit` инициализируется `None` до `try`. Это предотвращает `UnboundLocalError` в случае, если `begin_commit()` сам выбрасывает исключение.

---

### 4. `footprints.py` — работа с компонентами

Модуль для поиска, чтения и изменения футпринтов (компонентов).

**Поиск:**
- `get_all(board)` → `list[FootprintInstance]` — все компоненты платы.
- `get_by_ref(board, ref)` → `FootprintInstance` или `None` — поиск по refdes (например, `"C5"`). Регистрозависимо.

**Чтение:**
- `get_reference(fp)`, `get_value(fp)`, `get_footprint_name(fp)` — строки.
- `get_position_mm(fp)` → `(x_mm, y_mm)`.
- `get_angle_deg(fp)` → `float` (градусы).
- `get_layer(fp)` → `BoardLayer` (константа).
- `is_back(fp)` → `bool` — `True`, если компонент на нижней стороне.
- `get_bounding_box_mm(board, fp)` → `(width_mm, height_mm)` или `None` — габариты отдельного компонента (без текста).
- `get_bounding_boxes_mm(board, footprints)` → `list[Optional[tuple]]` — батч-запрос для нескольких компонентов (эффективнее, чем по одному).

**Изменение (локальное, требует `update_items` внутри коммита):**
- `set_position(fp, x_mm, y_mm)` — меняет позицию в локальном объекте.
- `set_angle_deg(fp, angle_deg)` — меняет угол.

**Настоящий переворот:**
- `flip_selected(kicad, board, footprints)` — выполняет GUI-флип для переданных футпринтов. **После вызова необходимо перечитать плату**, потому что локальные объекты устарели.

---

### 5. `pads.py` — площадки (пады)

Работа с падами **в контексте уже известного футпринта**.

- `get_all(fp)` → `list[Pad]` — все пады футпринта. Извлекаются из `fp.definition.items` (без обращения к API и геометрического поиска).
- `get_by_number(fp, number)` → `Pad` или `None` — поиск по номеру (например, `"1"`).
- `get_position_mm(pad)` → `(x_mm, y_mm)` — абсолютная позиция (kipy учитывает поворот футпринта).
- `get_net_name(pad)` → `str` — имя цепи или пустая строка, если не подключена.
- `get_size_mm(pad)` → `(width_mm, height_mm)` или `None` — размеры пада (берётся первый медный слой).
- `get_angle_deg(pad)` → `float` — собственный угол падстека (уже синхронизирован с футпринтом).

---

### 6. `nets.py` — цепи (Net)

- `get_all(board)` → `list[Net]` — все цепи.
- `get_by_name(board, name)` → `Net` или `None` — поиск по имени.
- `build_net_map(board)` → `dict[int, str]` — словарь {код: имя}. **Примечание:** свойство `Net.code` помечено в `kipy` как устаревшее, поэтому использование этой функции не рекомендуется в продуктивном коде; она сохранена только для отладки.

---

### 7. `vias.py` — переходные отверстия

- `make(position_mm, net, drill_mm=0.3, diameter_mm=0.6)` → `Via` — создаёт объект `Via` (пока не на плате). После этого нужно вызвать `board.create_items([via])` внутри транзакции.
- `remove_by_id(board, uuid_str)` — удаляет объект по UUID. **Важно:** внутри используется `board.remove_items_by_id([KIID])`, где `KIID` — это объект, а не строка. Функция корректно преобразует строку.

---

### 8. `zones.py` — зоны (в том числе Rule Areas)

- `get_by_name(board, name)` → `Zone` или `None` — поиск зоны по имени.
- `get_boundary_points(zone)` → `list[Vector2]` — точки контура (только узлы типа `has_point`; дуги игнорируются). Для большинства прямоугольных/полигональных зон этого достаточно.

---

### 9. `project.py` — путь к проекту и схеме

- `get_project_path(board)` → `str` или `None` — каталог проекта.
- `get_project_name(board)` → `str` или `None` — имя проекта.
- `get_schematic_path(board)` → `str` или `None` — полный путь к файлу `.kicad_sch` (собирается из пути и имени).

---

### 10. `selection.py` — работа с выделением

Модуль для анализа текущего выделения в PCB-редакторе.

- `get_selected_uuids(board)` → `set[str]` — UUID всех выделенных объектов. **Критично:** для групп (`Group`) используется `group.proto.items`, а не пустой `group.items` (локальный кеш).
- `get_selected_footprints(board)` → `list[FootprintInstance]` — только компоненты из выделения (другие типы игнорируются).
- `describe_selected(board)` → `list[dict]` — подробное описание каждого выделенного компонента: refdes, позиция, угол, размеры, список падов с их цепями и координатами. Удобно для диагностики и отладки.

---

## Взаимосвязи и типичный сценарий использования

1. **Подключение** через `kicad_client.connect()`.
2. **Получение платы** через `kicad_client.get_board(kicad)`.
3. **Чтение** – используйте `footprints.get_by_ref`, `pads.get_all`, `nets.get_by_name` и т.д.
4. **Изменение** – всегда внутри транзакции:
   ```python
   commit = board.begin_commit()
   try:
       # локально изменяем объекты (set_position, set_angle, создаём via)
       board.update_items([modified_objects])
       board.create_items([new_via])
       board.push_commit(commit, "my change")
   except Exception:
       board.drop_commit(commit)
       raise
   ```
5. **После GUI-действий** (например, `footprints.flip_selected`) обязательно вызовите `board = kicad_client.get_board(kicad)` или `board.refresh()`, чтобы обновить локальные объекты.

---

## Примечания по безопасности и производительности

- **Таймаут** увеличен до 20 секунд – достаточен для массовых операций (сотни объектов).
- **Батч-запросы** – используйте `get_bounding_boxes_mm` вместо циклического вызова `get_bounding_box_mm` для экономии времени.
- **Никакого геометрического маппинга** падов к футпринтам – мы всегда получаем пады через `definition.items` известного футпринта, что надёжнее и быстрее.
- **Опасные операции** (флип, создание/удаление объектов) выполняются только в тестах с флагом `dangerous`, но сам `core_api` не содержит логики защиты – она возложена на раннер.

# tests/decap_tools — утилиты и тесты для работы с декаплинговыми конденсаторами и переходными отверстиями

Каталог `tests/decap_tools/` содержит **опасные** диагностические и вспомогательные тесты, предназначенные для проверки и демонстрации IPC-взаимодействия с KiCad при выполнении типовых операций с компонентами (перемещение, переворот) и переходными отверстиями (создание, удаление, автоматическая расстановка). Эти тесты являются частью общей системы раннера, зарегистрированы в суите `decap_tools` и требуют осознанного включения (`enabled: true`) в конфиге из-за их способности изменять реальную плату.

## 1. `test_create_one_via.py` — минимальный тест создания/удаления via

**Назначение:** проверка базовых операций `create_items()` и `remove_items_by_id()` на примере одного переходного отверстия. Тест создаёт via рядом с центром указанного компонента, сохраняет его UUID в JSON-файл (`.last_test_via.json`) для последующего удаления без необходимости копировать идентификатор вручную.

**Функция:**  
`run_test(logger, kicad, board, ref=None, offset_mm=1.2, net="GND", drill_mm=0.3, diameter_mm=0.6, remove=False, remove_uuid=None, **params) -> bool`

- **Параметры:**
  - `ref` — refdes компонента (обязателен, если `remove=False`).
  - `offset_mm` — смещение по X от центра компонента (мм).
  - `net` — имя цепи (по умолчанию `"GND"`).
  - `drill_mm` и `diameter_mm` — параметры via.
  - `remove` — если `True`, то вместо создания удаляет via (используя сохранённый UUID или переданный `remove_uuid`).
- **Логика:**
  - При удалении: если `remove_uuid` не задан, читает `id` из JSON-файла.
  - При создании: находит компонент, цепь, строит via, выполняет транзакцию `begin_commit` → `create_items` → `push_commit`, сохраняет `id` в JSON.
- **Использует:** `core_api.footprints`, `core_api.nets`, `core_api.vias`, `core_api.board` (транзакции), `runner.step_helper.call_step`.

---

## 2. `test_flip_one_cap.py` — переворот компонента (тумблер)

**Назначение:** проверка настоящего переворота компонента (GUI-действие `pcbnew.InteractiveEdit.flip`), который зеркалирует площадки и шёлкографию, в отличие от простого изменения поля `.layer`. Тест полезен как быстрая самостоятельная проверка работоспособности флипа в текущей версии KiCad.

**Функция:**  
`run_test(logger, kicad, board, ref=None, **params) -> bool`

- **Параметры:** `ref` — refdes компонента.
- **Логика:**
  - Логирует состояние компонента до флипа (слой, позиция, угол).
  - Вызывает `footprints.flip_selected(kicad, board, [target])`.
  - Перечитывает компонент и логирует состояние после.
  - Возвращает `True`, если слой изменился на `BL_B_Cu` (или хотя бы изменился по сравнению с исходным).
- **Важно:** флип — операция-тумблер; повторный вызов вернёт компонент обратно.
- **Использует:** `core_api.footprints` (в частности `flip_selected`).

---

## 3. `test_move_one_cap.py` — сдвиг компонента по оси X

**Назначение:** изолированный тест IPC-записи — сдвиг одного компонента на заданное расстояние. Используется для диагностики зависаний транзакций (`begin_commit`) и проверки механизма `update_items`.

**Функция:**  
`run_test(logger, kicad, board, ref=None, delta_mm=1.0, revert=False, **params) -> bool`

- **Параметры:** `ref` — refdes компонента; `delta_mm` — величина сдвига (мм); `revert` — если `True`, сдвиг выполняется в обратную сторону (`-delta_mm`).
- **Логика:**
  - Вычисляет новую позицию.
  - Открывает транзакцию, обновляет позицию через `update_items`, фиксирует.
- **Использует:** `core_api.footprints` (чтение и изменение позиции), `core_api.geometry.vec_mm`, `runner.step_helper.call_step`.

---

## 4. `test_via_placer.py` — автоматическая расстановка via возле GND-падов

**Назначение:** размещение переходных отверстий возле GND-падов заданных компонентов с учётом трёх стратегий отступа (`center`, `edge`, `courtyard`), поиском свободного места с учётом зон запрета (keepout) от других компонентов и существующих via.

**Функция:**  
`run_test(logger, kicad, board, gnd_net_name="GND", via=None, components=None, dry_run=False, **params) -> bool`

- **Параметры:**
  - `gnd_net_name` — имя цепи (по умолчанию `"GND"`).
  - `via` — словарь параметров via:
    - `offset_from` — `"center"`, `"edge"` или `"courtyard"`.
    - `offset_mm` — базовый отступ (мм).
    - `angle_deg` — предпочтительный угол (если не задан, вычисляется автоматически).
    - `drill_mm`, `diameter_mm` — размеры via.
    - `keepout_clearance_mm` — запас для зон запрета.
    - `search_step_mm`, `search_max_radius_mm`, `search_n_directions` — параметры поиска свободной точки.
  - `components` — список словарей с полями `ref` (обязательно), `offset_mm`, `angle_deg` (переопределяют глобальные).
  - `dry_run` — если `True`, только логирует, не создаёт via.
- **Логика:**
  - Для каждого компонента:
    - Находит футпринт и GND-пад.
    - Определяет стратегию отступа.
    - Вычисляет идеальную позицию via с учётом поворота пада и Courtyard (если выбран).
    - Строит зоны запрета из bounding box’ов остальных компонентов и всех via.
    - Ищет свободную точку в окрестности идеальной позиции с учётом предпочтительного направления.
    - Создаёт объект via.
  - В конце выполняет одну транзакцию на создание всех via (или ничего, если `dry_run`).
- **Вспомогательные функции внутри файла:**
  - `intersect_ray_with_rotated_rect` — расчёт пересечения луча с повёрнутым прямоугольником (для `edge`).
  - `get_courtyard_polygon` — извлечение точек полигона Courtyard из `fp.definition.items` (исправлено — теперь корректно обрабатывает `BoardRectangle` и `BoardPolygon`).
  - `compute_ideal_position` — вычисление идеальной позиции по выбранной стратегии.
- **Использует:** `core_api.footprints`, `core_api.pads`, `core_api.nets`, `core_api.vias`, `runner.step_helper.call_step`, локальные модули `geometry.boundary` и `geometry.keepout`.

---

## 5. `geometry/boundary.py` — утилиты работы с полигональными границами

**Назначение:** набор функций для работы с полигонами (Courtyard, зоны) — вычисление пересечения луча с полигоном, ближайшей точки на полигоне и нормали наружу. Используется в `test_via_placer` для стратегии `courtyard`.

**Ключевые функции:**
- `polyline_points(polyline)` — извлекает точки из `PolyLine`.
- `_ray_segment_intersection(ox, oy, dx, dy, x1, y1, x2, y2)` — пересечение луча и отрезка.
- `ray_boundary_distance(center, target, boundary_pts)` — расстояние от `center` до пересечения луча с полигоном.
- `polygon_signed_area(polygon)` — знаковая площадь (для определения ориентации обхода).
- `closest_point_on_polygon(point, polygon)` — ближайшая точка на полигоне и нормаль наружу. **Исправлено:** направление нормали определяется один раз для всего полигона через знаковую площадь, что корректно работает даже в пограничных случаях, когда точка находится вне полигона впритык.

**Примечание:** код скопирован из проекта `KiCadDecapPlacer` и пока является дублированием, но планируется консолидировать в общее место.

## 6. `geometry/keepout.py` — зоны запрета и поиск свободной точки

**Назначение:** построение прямоугольных зон запрета (keepout) на основе bounding box’ов объектов и поиск свободной точки для размещения via с учётом предпочтительного направления.

**Класс `Rect`:**
- Простой AABB-прямоугольник в нанометрах.
- Конструктор от bounding box’а (`from_bbox`) с запасом `clearance`.
- Аппроксимация окружности квадратом (`from_circle`).
- Проверка пересечения (`intersects`).

**Функции:**
- `point_is_clear(point, via_radius, keepout)` — проверяет, не пересекает ли окружность via радиусом `via_radius` ни один прямоугольник из `keepout`.
- `build_keepout(bboxes, clearance_mm, mm_per_unit)` — строит список `Rect` из списка bounding box’ов с запасом.
- `find_free_point(ideal, keepout, via_radius, preferred_direction, step_mm, max_radius_mm, mm_per_unit, n_directions)` — ищет ближайшую к `ideal` свободную точку, расширяясь концентрическими кольцами. На каждом кольце сначала проверяется предпочтительное направление, затем равномерно распределённые направления. Возвращает `Vector2` или `None`, если место не найдено в пределах `max_radius_mm`.

**Используется:** в `test_via_placer` для финального позиционирования via.

## 1. `test_flip_then_update_items.py` – регрессия на «стухший» объект после флипа

**Файл:** `test_flip_then_update_items.py`  
**Имя теста:** `mutating_flip_then_update_items`

### Назначение
Проверка, что после GUI-флипа (`pcbnew.InteractiveEdit.flip`) нельзя использовать старый локальный объект `Footprint` в `update_items()`, потому что это тихо откатит изменения слоя. Тест демонстрирует **неправильный** путь (A) и **правильный** путь (B) с перечитыванием футпринта после флипа.

### Функция
```python
run_test(logger, kicad, board, ref=None, **params) -> bool
```

**Параметры:** `ref` – refdes тестового компонента (обязателен).

### Логика
1. Сохраняет исходные слой и позицию.
2. **Вариант A (неправильный):**
   - Выделяет компонент, выполняет `run_action("pcbnew.InteractiveEdit.flip")`.
   - Открывает транзакцию, модифицирует **старый** объект `target.position` (сдвиг на 0.5 мм).
   - Вызывает `board.update_items([target])` – это откатывает слой.
   - Проверяет, вернулся ли слой к исходному – если да, баг воспроизведён.
3. **Вариант B (правильный):**
   - Вызывает `footprints.flip_selected(kicad, board, [after_a])` (это тоже GUI-флип, но через обёртку).
   - **Перечитывает** футпринт (`get_by_ref`) – получает свежий объект.
   - Открывает транзакцию, восстанавливает исходную позицию и фиксирует.
   - Проверяет, что слой стал исходным (второй флип вернул его обратно).
4. Логирует результаты.

### Важные замечания
- Тест **мутирует** компонент (флип + сдвиг), но в конце восстанавливает исходное состояние через вариант B.
- Демонстрирует критическую ошибку: `update_items` со старым объектом после флипа – тихий no-op для слоя, но сдвиг позиции применяется.
- Требует тестовый компонент (не боевой).

## 2. `test_footprint_rotation_linkage.py` – проверка бага #21655 (разрыв связи symbol ↔ footprint)

**Файл:** `test_footprint_rotation_linkage.py`  
**Имя теста:** `mutating_footprint_rotation_linkage`

### Назначение
Проверка косвенных признаков целостности связи между символом схемы и футпринтом после поворота через IPC. Прямая проверка невозможна, так как IPC не даёт доступа к редактору схем. Тест сравнивает снимки до и после поворота: `reference`, `value`, имя футпринта, `net` на каждом паде.

### Функция
```python
run_test(logger, kicad, board, ref=None, angle_deg=45.0, auto_revert=True, **params) -> bool
```

**Параметры:**
- `ref` – refdes компонента (обязателен).
- `angle_deg` – величина поворота в градусах (по умолчанию 45°).
- `auto_revert` – если `True` (по умолчанию), тест возвращает угол обратно после проверки.

### Логика
1. Создаёт `_snapshot(fp)` – словарь с `ref`, `value`, `footprint_name`, `pad_count`, `pads` (отсортированный список `(номер, net)`).
2. Поворачивает компонент на `angle_deg` через `board.update_items` (внутри транзакции).
3. Перечитывает футпринт и делает снимок после.
4. Сравнивает снимки – если они идентичны (кроме угла), тест считает, что на PCB-стороне связность не пострадала.
5. Если `auto_revert=True`, выполняет обратный поворот (`-angle_deg`).
6. Логирует результат.

### Важные замечания
- Тест **не доказывает** отсутствие бага #21655 на стороне схемы, но показывает, что `net` на падах не потерялись.
- По умолчанию плата **не остаётся изменённой** благодаря `auto_revert`.
- Если `auto_revert=False`, компонент остаётся повёрнутым для ручной проверки в Schematic Editor.

## 3. `test_pad_ownership_comparison.py` – сравнение двух способов получения падов

**Файл:** `test_pad_ownership_comparison.py`  
**Имя теста:** `mutating_pad_ownership_comparison`  
**Опасность:** `False` (только чтение)

### Назначение
Проверка, нужен ли геометрический маппинг для сопоставления падов с футпринтами. ## 4. `test_refill_zones_busy_repro.py` – воспроизведение «вечного busy» после `refill_zones(block=False)`

**Файл:** `test_refill_zones_busy_repro.py`  
**Имя теста:** `mutating_refill_zones_busy_repro`  
**Опасность:** **очень высокая** – может привести к невосстанавливаемому состоянию платы, требующему перезапуска KiCad.

### Назначение
Проверка гипотезы, что вызов `board.refill_zones(block=False, max_poll_seconds=1)` оставляет незавершённый асинхронный джоб, который «заклинивает» плату: все последующие вызовы (`get_footprints`, `get_tracks` и т.д.) возвращают `busy` навсегда.

### Функция
```python
run_test(logger, kicad, board, **params) -> bool
```

### Логика
1. **Шаг 1 (baseline):** вызывает `get_footprints`, `get_tracks`, `get_nets` – проверяет, что всё работает до операции.
2. **Шаг 2:** выполняет `board.refill_zones(block=False, max_poll_seconds=1)` – **точно как в оригинальном инциденте**.
3. **Шаг 3 (сразу после):** повторяет content-вызовы и фиксирует, сломались ли они.
4. **Шаг 4 (повторные попытки):** делает до 6 попыток с паузами по 5 секунд, чтобы увидеть, отпускает ли блокировка сама.
5. Логирует результат.

### Важные замечания
- **Самый опасный тест.** Если гипотеза подтвердится, плата остаётся в «битом» состоянии, и **единственный выход** – полный перезапуск приложения KiCad.
- Запускать **только** на тестовой плате, не в середине важной сессии.
- Тест не выполняет `refill_zones` с `block=True`, потому что это блокирует IPC-соединение на длительное время и не является воспроизведением исходного инцидента.

- `core_api.kicad_client`

# tests/static — статические контрактные проверки библиотеки `kipy`

Каталог `tests/static/` содержит **один файл** – `test_kipy_contract.py`, в котором реализованы **статические контрактные тесты**. Эти тесты **не требуют запущенного KiCad** и не открывают IPC-соединение – они анализируют только типы, атрибуты, сигнатуры методов и поведение объектов **прямо во время импорта** библиотеки `kicad-python` (kipy). Они служат для раннего обнаружения несовместимостей API, которые могут привести к:

- **тихим no-op операциям** (например, присваивание атрибуту копии объекта);
- **исключениям** (например, передача неправильного типа в сеттер);
- **потере данных** (например, игнорирование угла поворота или слоя компонента);
- использованию **устаревших или нестабильных** API.

Все тесты зарегистрированы в суите `static`, помечены `dangerous=False`, **не используют** `needs_kicad` (по умолчанию `False`), поэтому раннер не передаёт им объекты `kicad`/`board`. Они выполняются быстро и безопасно, идеально подходят для CI/CD.

---

## Файл: `test_kipy_contract.py`

Этот файл содержит 12 отдельных проверок, каждая из которых эмулирует конкретный баг, найденный в инструментах `KiCadTemplateCloner` или `KiCadDecapPlacer`, или проверяет документацию/поведение `kipy` на соответствие реальности.

Все проверки используют модуль `inspect` для анализа сигнатур и docstring, а также создают экземпляры классов и проверяют наличие атрибутов/методов.

## Список тестов (функций)

### 1. `static_via_drill_attribute_name`

**Имя теста:** `static_via_drill_attribute_name`

**Назначение:** Проверка, что у объекта `Via` есть плоский атрибут `drill_diameter`, а не вложенный `drill.diameter` (как ошибочно использовалось в `KiCadTemplateCloner`).

**Логика:**
- Создаёт экземпляр `Via`.
- Проверяет `hasattr(v, "drill")` (должно быть `False`) и `hasattr(v, "drill_diameter")` (должно быть `True`).
- Логирует результат.

**Источник бага:** `extractor.py:99`, `applier.py:20` – обращение к `via.drill.diameter`.

**Ожидание:** `has_flat_drill=True`, `has_nested_drill=False`.

### 4. `static_getter_returns_copy_not_reference`

**Имя теста:** `static_getter_returns_copy_not_reference`

**Назначение:** **Критическая проверка**, что геттеры типа `.position` возвращают **копию** объекта, а не ссылку. Присваивание атрибуту `.x` на этой копии – тихий no-op, позиция не меняется. Это самый важный баг, найденный в `applier.py`.

**Логика:**
- Создаёт `FootprintInstance`, устанавливает `position = Vector2(1_000_000, 2_000_000)`.
- Сохраняет `before_x = fp.position.x`.
- Выполняет `fp.position.x = 99_000_000` (как в `applier.py`).
- Читает `after_x = fp.position.x`.
- Проверяет, что `after_x == before_x` и `after_x != 99_000_000` – это доказывает no-op.

**Источник бага:** `applier.py:17-19, 73-74` – `fp.position.x = ...` и `via.position.x = ...`.

**Ожидание:** `is_noop=True` – подтверждение, что такой код не работает.

**Исправление:** переприсваивать объект целиком: `fp.position = Vector2(...)`.

# tests/toy — тесты-заглушки для проверки механики раннера

Каталог `tests/toy/` содержит **тесты-заглушки** (dummy-тесты), предназначенные исключительно для проверки и отладки **самого раннера** (`runner`), а не для взаимодействия с KiCad. Они не требуют запущенного KiCad, не используют IPC и не модифицируют плату. Эти тесты помогают убедиться, что:

- конфигурация загружается и применяется;
- отбор тестов (по суите, по имени, с учётом `enabled`/`dangerous`) работает корректно;
- параметры подставляются через `resolve_params`;
- исключения перехватываются и логируются;
- итоговый отчёт формируется правильно.

Все тесты зарегистрированы в суите `toy`, по умолчанию `dangerous=False` (кроме одного), `needs_kicad=False` (не требуют соединения). В продуктивной среде эта суита обычно **отключена** (`enabled: false`), чтобы не засорять логи.

---

## Структура каталога

```
tests/toy/
├── __init__.py
└── dummy_tests.py
```

---

## Файл: `dummy_tests.py`

Содержит 5 тестов-заглушек, покрывающих различные сценарии выполнения.

---

### 1. `dummy_ok` – успешный тест

**Функция:** `run_dummy_ok(logger, **params) -> bool`

**Поведение:** Всегда возвращает `True`. Логирует полученные параметры.

**Назначение:** Проверка успешного прохождения теста – раннер должен засчитать его как `PASS`.

---

### 2. `dummy_fail` – провальный тест

**Функция:** `run_dummy_fail(logger, **params) -> bool`

**Поведение:** Всегда возвращает `False`. Логирует сообщение.

**Назначение:** Проверка обработки неуспешного теста – раннер должен засчитать его как `FAIL` и включить в итоговую статистику.

---

### 3. `dummy_raises` – тест, бросающий исключение

**Функция:** `run_dummy_raises(logger, **params) -> bool`

**Поведение:** Бросает `RuntimeError` с сообщением "нарочная ошибка для проверки обработки исключений раннером".

**Назначение:** Проверка, что раннер перехватывает исключение, логирует его (с traceback), считает тест проваленным и продолжает выполнение остальных тестов.

---

### 4. `dummy_needs_params` – тест, требующий параметры

**Функция:** `run_dummy_needs_params(logger, ref=None, pad=None, **params) -> bool`

**Поведение:** Возвращает `True`, только если `ref` и `pad` не равны `None`. Логирует переданные значения.

**Назначение:** Проверка механизма `resolve_params` – что параметры из профиля платы и теста правильно подставляются и переопределяются.

---

### 5. `dummy_dangerous` – опасный (по флагу) тест

**Функция:** `run_dummy_dangerous(logger, **params) -> bool`

**Регистрация:** `@register("dummy_dangerous", suite="toy", dangerous=True)`

**Поведение:** Всегда возвращает `True`, но логирует предупреждение: _"этот тест реально выполнился — если видите это без явного enabled:true в конфиге, значит защита не сработала!"_

**Назначение:** Проверка, что опасные тесты **не запускаются** без явного `enabled: true` в конфиге, даже если они включены в суиту. Это критическая защита от случайного выполнения опасных мутаций.

## Общая характеристика суиты `toy`

| Тест | Возвращает | Исключение | Параметры | Опасный |
|------|------------|------------|-----------|---------|
| `dummy_ok` | `True` | нет | нет | нет |
| `dummy_fail` | `False` | нет | нет | нет |
| `dummy_raises` | – | `RuntimeError` | нет | нет |
| `dummy_needs_params` | `True` при наличии `ref` и `pad` | нет | `ref`, `pad` | нет |
| `dummy_dangerous` | `True` | нет | нет | **да** |

Все тесты используют `needs_kicad=False` (по умолчанию), поэтому раннер **не передаёт** им объекты `kicad`/`board`.

## Запуск и настройка

Суита `toy` обычно **выключена** в продакшен-конфигах (`enabled: false`), но может быть включена для отладки раннера.

Пример включения в YAML:

```yaml
suites:
  toy:
    enabled: true
    tests:
      dummy_ok:
        enabled: true
      dummy_fail:
        enabled: true
      dummy_raises:
        enabled: true
      dummy_needs_params:
        enabled: true
        params:
          ref: "C1"
          pad: "1"
      dummy_dangerous:
        enabled: false   # опасный – должен быть явно включён
```

Запуск:

```bash
python -m runner.run --config config.yaml --suite toy
```

Или отдельный тест:

```bash
python -m runner.run --config config.yaml --test dummy_needs_params
```

Для `dummy_needs_params` параметры можно передать через конфиг (как выше) или через `--board` профиль, если в профиле есть `ref` и `pad`.
