# IPC Тесты для KiCad 10

Набор модульных тестов для проверки работоспособности IPC API KiCad 10 через библиотеку `kipy`.

## Структура
- `core.py` – подключение и логирование
- `board_utils.py`, `component_utils.py`, `net_utils.py`, `project_utils.py` – утилиты
- `cli_utils.py` – работа с kicad-cli
- `tests/` – сами тесты
- `logs/` – каталог для логов (создаётся автоматически)

## Запуск
```bash
python run_all_tests.py
```

## Изменения от 2026-07-12

По логам первого прогона (`logs/test.log`) нашлись и исправлены реальные баги:

1. **Главная причина `"KiCad is busy"` на большинстве тестов**: каждый
   `test_*.py` открывал НОВОЕ IPC-подключение через `get_kicad_board()`, ни
   одно из них не закрывалось (`kipy.KiCad` в 0.7.1 не имеет `close()`), и
   к моменту тяжёлых тестов на плате скапливалось 7-8 незакрытых сокетов.
   `core.get_kicad_board()` теперь кэширует одно соединение на процесс и
   переиспользует его (с автоматическим переподключением, если оно
   протухло). Проверяется тестом `test_connection_reuse.py`.
2. `test_full_api.py`: `UnboundLocalError` из-за необъявленных переменных
   при исключении внутри `try`, и неверный вызов `board.get_items(types=
   0xFFFFFFFF)` — `types` это значения enum `KiCadObjectType`, а не битовая
   маска.
3. `component_utils.get_pads()` обращался к несуществующему
   `footprint.definition.pads` (в 0.7.1 это `definition.items`,
   отфильтрованный по `isinstance(item, Pad)`).
4. `board_utils.get_board_info()` вызывал несуществующий `board.get_size()`
   — размер платы теперь считается через bounding box контура `Edge.Cuts`.
5. `cli_utils.export_netlist()`: команда `kicad-cli` была
   `schematic export netlist` (не существует) с форматом `xml` (тоже не
   существует) — исправлено на `sch export netlist --format kicadxml`.
   Плюс ошибки `kicad-cli` (stdout/stderr) теперь логируются, а не
   проглатываются.

Во всех утилитах ошибки больше не глушатся молча (`except: return []`) —
они логируются через параметр `logger`, что раньше маскировало настоящие
причины под безобидным «Нет компонентов».

Добавлено подробное логирование: `core.call_ipc()` — обёртка для вызова
любого метода IPC с таймингом (мс) и единообразным `[OK]`/`[ERR]` в логе;
`test_all.py` теперь пишет время каждого теста и общее время прогона.

## Обновление от прогона 2026-07-12 14:06-14:07

Фикс с переиспользованием соединения подтверждён (`test_connection_reuse`
проходит, повторные `get_kicad_board()` — 0.0 мс). НО гипотеза про
накопление соединений как причину "busy" — **опровергнута**: тайминги
показали, что busy возвращается мгновенно (0.1-0.5 мс), это не таймаут.

Новый паттерн: падают мгновенным busy только вызовы, трогающие содержимое
платы (`get_footprints`, `get_tracks`, `get_vias`, `get_zones`, `get_pads`,
`get_text`, `get_shapes`, `get_dimensions`, `get_groups`, `get_items`,
`begin_commit`, `save`), а метаданные (`get_nets`, `get_project`,
`get_title_block_info`, `get_copper_layer_count`, `get_enabled_layers`,
`get_active_layer`, `get_selection`) проходят нормально — в рамках ОДНОГО
и того же соединения. По логу видно, что это переключение произошло между
двумя прогонами в один и тот же (не перезапущенный) сеанс KiCad — то есть
похоже на состояние живого GUI PCB-редактора (открытый диалог, активный
инструмент, редактор не в фокусе), а не на серверную перегрузку.

Добавлен диагностический `test_footprints_probe.py` — несколько попыток
`get_footprints()` с паузами, чтобы отличить "временно занят" от
"подвис насовсем, пока не потрогать GUI руками".

## Обновление от прогона 2026-07-12 14:16 (проба на 6 попыток/12 сек)

Проба (`test_footprints_probe`) показала: busy держится стабильно все 6
попыток за 12 секунд, ничего не отпускает само. Пользователь подтвердил,
что в окне PCB Editor в этот момент визуально ничего не происходит (нет
диалогов, нет активного инструмента) — значит гипотеза "что-то в GUI"
неверна в буквальном смысле "занят прямо сейчас".

Рабочая гипотеза теперь другая: где-то между 13:47 и 13:49 в самом первом
(ещё непатченом) прогоне `get_zones()` один раз сработал, и следом
вызвался `board.refill_zones(block=False, max_poll_seconds=1)` —
незавершённая асинхронная заливка зон без дальнейшего опроса статуса
могла оставить плату в постоянном "занято" для операций с содержимым, не
трогая при этом чтение метаданных (nets/project/layers). Это совпадает по
времени с моментом поломки и держится с тех пор в одной и той же
долгоживущей сессии KiCad.

Проверяется одним действием: **полностью закрыть и заново открыть само
приложение KiCad** (не только перезапустить скрипт), затем сразу прогнать
тесты первым делом. Если busy пропадёт — гипотеза подтверждена.

Также на всякий случай `refill_zones` в `test_full_api.py` переведён на
`block=True` — больше не оставляем асинхронный джоб висеть незавершённым.

`cli_utils.py`: жёстко прописанный путь к `kicad-cli.exe` заменён на
`find_kicad_cli()` — сначала `KICAD_CLI_PATH`, затем несколько типичных
путей установки на Windows (включая `%LOCALAPPDATA%\Programs\KiCad\10.0`),
затем поиск в PATH.
