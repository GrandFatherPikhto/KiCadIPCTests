# README для `ipc_tests` — тестовый фреймворк для IPC KiCad 10

Этот пакет содержит **набор интеграционных, регрессионных и статических тестов** для проверки работы IPC-интерфейса KiCad 10 через библиотеку `kicad-python` (kipy) версии 0.7.1.

Тесты предназначены для:
- проверки доступности и корректности IPC-вызовов;
- выявления и воспроизведения специфических багов (например, потеря данных после флипа, no-op присваивания, busy-состояния);
- диагностики проблем на живом KiCad;
- валидации контрактов библиотеки `kipy` (без запуска KiCad).

| `tests/*.py` | **Безопасные тесты** – только чтение данных с платы, не изменяют её. |

## ⚙️ Требования

- KiCad 10.0.4 (или новее) с открытым PCB-редактором
- Python 3.8+
- Библиотека `kicad-python` (kipy) версии **0.7.1**
- Для CLI-тестов – `kicad-cli.exe` (должен быть в PATH или указан через переменную `KICAD_CLI_PATH`)

## 🔧 Настройка

### Путь к IPC-сокету

По умолчанию используется сокет:

```
ipc://C:\Users\<username>\AppData\Local\Temp\kicad\api.sock
```

Если ваш путь отличается, укажите его в переменной окружения:

```bash
set KICAD_IPC_SOCKET=ipc://C:\путь\к\api.sock
```

или передайте параметр `socket_path` в `get_kicad_board()`.

### Логирование

Все логи пишутся в файл `logs/test.log` с ротацией (максимум 5 МБ, 3 бэкапа).  
Уровень логирования:
- консоль: `INFO` (можно изменить, передав `console_level=logging.DEBUG`)
- файл: `DEBUG` (содержит тайминги каждого IPC-вызова)

Логирование настраивается вызовом `setup_logging()` (по умолчанию вызывается в каждом тесте).

## 🚀 Запуск тестов

### Запуск всех безопасных тестов (рекомендуется)

```bash
python -m ipc_tests.tests.test_all
```

Это выполнит все тесты из папки `tests/` в порядке, указанном в `test_all.py`.

### Запуск отдельного безопасного теста

```bash
python -m ipc_tests.tests.test_connection
python -m ipc_tests.tests.test_board
python -m ipc_tests.tests.test_components
# и т.д.
```

### Статические контрактные тесты (без KiCad)

```bash
python -m ipc_tests.static.run_static_tests
```

### Мутирующие тесты (только на тестовых платах!)

```bash
# Тест флипа с последующим update_items (воспроизведение бага)
python -m ipc_tests.mutating.test_flip_then_update_items C401

# Тест поворота и проверки целостности данных
python -m ipc_tests.mutating.test_footprint_rotation_linkage C401 --angle-deg 45

# Воспроизведение busy после refill_zones(block=False)
python -m ipc_tests.mutating.test_refill_zones_busy_repro
```

**Внимание!** Мутирующие тесты **изменяют** плату (переворачивают компоненты, поворачивают, изменяют позицию). Запускайте их **только** на копии тестового проекта, а не на боевой плате.

## 🧠 Важные замечания

### Кэширование соединения
В `core.py` реализован глобальный кэш объекта `KiCad`. Это устраняет проблему накопления сокетов (раньше каждый тест создавал новое соединение, что приводило к busy-ошибкам). Кэш проверяется на живость через `board.get_project()` – если вызов падает, соединение пересоздаётся.

### Обёртка `call_ipc()`
Все вызовы IPC должны проходить через `call_ipc(logger, label, func, ...)`. Это гарантирует:
- единообразное логирование с таймингом;
- перехват исключений и их логирование;
- возврат `(result, ok)`, где `ok` – успешность вызова.

### Статические тесты и найденные баги
Многие тесты в `test_kipy_contract.py` были написаны после того, как реальные баги были обнаружены в production-коде (TemplateCloner, DecapPlacer). Они подтверждают, что:
- геттеры (`fp.position`, `via.net`) возвращают **копии**, поэтому изменение `.x` или `.name` – **no-op**;
- сеттер `fp.orientation` принимает только объект `Angle`, а не число;
- `Board.push_commit()` требует обязательный аргумент `commit`;
- у `Via` нет вложенного `drill` – только плоский `drill_diameter`.

### Мутирующие тесты и безопасность
Эти тесты **реально изменяют плату**. Перед запуском убедитесь, что:
- вы работаете с копией тестового проекта;
- у вас есть резервная копия `.kicad_pcb` (и `.kicad_sch`, если важна синхронизация);
- вы готовы к тому, что после некоторых тестов (особенно `test_refill_zones_busy_repro`) может потребоваться перезапуск KiCad.

### Интерпретация результатов
- Если все безопасные тесты проходят – IPC-соединение стабильно, основные вызовы работают.
- Если `test_footprints_probe` показывает постоянный busy – скорее всего, в GUI KiCad открыт диалог или активен инструмент, блокирующий модель. Закройте все диалоги и переключитесь на инструмент выбора (Select).
- Статические тесты должны всегда проходить (если версия `kipy` соответствует). Если какой-то падает – значит контракт библиотеки изменился, и production-код нужно пересмотреть.

## 🔗 Связь с `core_api`

Пакет `ipc_tests` **не зависит** от `core_api` – он использует `kipy` напрямую через свои утилиты.  
Однако обе группы тестов проверяют одни и те же сущности. Рекомендуется:
- использовать `core_api` в production-скриптах (он уже содержит все исправления и обёртки);
- использовать `ipc_tests` для регрессионного тестирования и диагностики.

Если вы вносите изменения в `core_api`, обязательно прогнать `ipc_tests` (особенно мутирующие), чтобы убедиться, что семантика не нарушена.

## ✍️ Как добавить новый тест

1. Создайте новый файл в `ipc_tests/tests/` (или в `mutating/` для опасных тестов).
2. Определите функцию `run_test(logger)`, которая возвращает `True` при успехе, иначе `False`.
3. Используйте `get_kicad_board(logger)` для получения объекта Board.
4. Все IPC-вызовы оборачивайте в `call_ipc(logger, "описание", func, ...)`.
5. Добавьте импорт и вызов вашего теста в `test_all.py` (в список `tests`).
6. Для статических тестов – добавьте функцию в `test_kipy_contract.py` и включите её в `ALL_TESTS`.

Пример:

```python
def run_test(logger):
    logger.info("=== МОЙ ТЕСТ ===")
    board = get_kicad_board(logger=logger)
    if board is None:
        return False
    result, ok = call_ipc(logger, "get_footprints", lambda: list(board.get_footprints()))
    if not ok:
        return False
    logger.info(f"Найдено {len(result)} компонентов")
    return True
```
